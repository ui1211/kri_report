# AGENTS.md

## Purpose
This project uses `redminewf` for Redmine operations from Codex.

## Rules for Codex

### Redmine Basic Rules
1. Redmine操作は必ずローカル仮想環境の実行ファイルを使うこと。
   - Windows: `.\.venv\Scripts\redminewf.exe`
   - Linux/macOS: `./.venv/bin/redminewf`
2. 実行ディレクトリはこのプロジェクトルート（`.env` と `.redmine.toml` がある場所）。
3. 優先順位は `CLI args > .redmine.toml > defaults`。
4. `allowed_*` が設定されている場合は、許可外の値を使わない（エラーで停止）。

### Redmine Ticket Rules
5. 担当者は原則 `me`。指示がある場合のみ対象ユーザーを指定する。
6. チケット作成時は期限 (`--due-date`) を必須指定し、開始日 (`--start-date`) も可能な限り指定する。
7. チケット名は原則 `【カテゴリ名】端的な要約` の形式にする。
8. チケット名に `【カテゴリ名】` の指定がある場合は、その指定に従う。
9. 親チケットがある場合、`【カテゴリ名】` は原則として親チケットに従う。
10. `【カテゴリ名】` の指定がない場合は、タスク内容から次のいずれかを選ぶ。
   - `開発`
   - `検証`
   - `企画`
   - `調査`
   - `改修`
   - `バグ修正`
11. カテゴリ名は今後拡張されうる前提で扱い、開発実装以外の企画・運用タスクにも対応する。

### Redmine Completion Rules
12. 完了時は以下を必ず実施する。
   - 進捗率を `100%` に更新
   - ステータスを `終了` に更新
   - 適切な完了コメントを入力

### Report Rules
13. チケット作業終了後は `report/template.md` を読み込み、`report/{#チケット番号}_{チケット名}.md` 形式でレポートを作成する。
14. レポートには少なくとも `概要` と `動作検証結果レポート` を記載し、コミット判断に必要な確認結果を残す。
15. レポートには、可能な限り実際に実施した動作確認結果を記載する。

### Git Commit Rules
16. Gitのコミットコメントは `[カテゴリ名]要約` の形式とし、要約は100文字以内で変更内容が明確に分かるように記載する。
17. コミットコメントのカテゴリ名は次を使用する。
   - `[add]`: 新規ファイル作成や機能をゼロから追加した場合
   - `[modify]`: 既存機能の変更や仕様変更に対応した場合
   - `[fix]`: 不具合修正やタイポ修正の場合
   - `[doc]`: README等のドキュメント修正・更新の場合
18. コミットコメント例: `[add] スタート画面のボタン作成`

### Security Rules
19. `.env` / `.redmine.toml` は機密として扱い、コミットしない。
