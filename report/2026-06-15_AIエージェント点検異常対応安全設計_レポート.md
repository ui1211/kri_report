# レポート

## 1. メタ情報
- 作成日時: 2026-06-15
- 入力概要: AI Agentを点検レポート、異常通知、再確認依頼、修繕判断補助に使う場合の安全設計を整理する。
- トピック数: 8

## 2. 概要
- 点検・異常対応にAIエージェントを使う場合、AIは文章生成だけでなく、Redmine起票、Slack通知、再確認依頼、外部API呼び出しまで実行し得るため、権限・承認・ログ・失敗時処理を最初から設計する必要がある。
- [n8n Agentの説明](https://docs.n8n.io/advanced-ai/examples/understand-agents/)では、Agentは問い合わせに応じてツールを選び、ツール呼び出しと応答評価のために複数回実行される。この性質は便利だが、点検業務では誤通知、過剰起票、機密漏えい、誤った修繕判断につながる。
- 安全設計の中心は、LLMに直接すべての権限を持たせず、入力検査、出力検査、ツール権限の分離、人承認、実行データの秘匿、監査ログ、Redmine/Slackの冪等処理をワークフロー側で担保することである。
- OWASPの[LLM Top 10 2025](https://genai.owasp.org/llm-top-10/)は、Prompt Injection、Sensitive Information Disclosure、Improper Output Handling、Excessive Agencyを主要リスクとして扱うため、点検・異常対応エージェントの設計基準として使える。

## 3. 現状
- 業界・領域全体の状況: n8nは[Advanced AI](https://docs.n8n.io/advanced-ai/)とAgent nodeを提供し、AIがツールを選択して外部処理を行うワークフローを構成できる。一方、OWASPは生成AI・LLMアプリケーションのリスクとして、Prompt Injection、過剰な代理実行、出力検証不足、機密情報漏えいを明示している。
- 実運用事例: 点検・異常対応では、画像異常検知やセンサー異常検知の結果を、Slack通知、Redmine起票、担当者確認、再取得、顧客報告へ接続する使い方が想定される。[Redmine REST API](https://www.redmine.org/projects/redmine/wiki/Rest_api)はIssueのCRUDを提供し、[Redmine Issues API](https://www.redmine.org/projects/redmine/wiki/Rest_Issues)はIssue作成、更新、コメント、添付を扱える。[Slack Incoming Webhooks](https://docs.slack.dev/messaging/sending-messages-using-incoming-webhooks/)はチャンネルへの投稿に使えるが、送信先チャンネルやユーザー名はアプリ設定を継承する制約がある。
- 運用上の前提: [n8n Wait node](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.wait/)はワークフローを一時停止し、Webhookやフォーム提出で再開できるため、人承認や現地確認待ちに使える。[Execution Data node](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.executiondata/)は実行メタデータを保存し、実行一覧で検索できる。[Execution data redaction](https://docs.n8n.io/workflows/executions/execution-data-redaction/)は本番実行データや手動実行データを秘匿できる。

## 4. 課題
- C1: AI AgentにRedmine起票、Slack通知、メール送信、API実行を直接許可すると、Prompt Injectionや誤推論によって過剰起票、誤通知、不要な修繕判断が発生し得る。
- C2: 点検画像、位置情報、顧客名、設備ID、APIキー、Redmineチケット内容などをAI入出力や実行ログに残すと、機密情報漏えいと権限越境のリスクがある。
- C3: AI出力をそのまま下流APIに渡すと、形式不備、過剰な自由記述、未検証URL、誤った担当者・期限・優先度が業務システムへ流れ込む。
- C4: 人確認なしに「異常確定」「補修必要」「顧客通知済み」まで進めると、AIの誤判定が現場判断や顧客対応に直結する。
- C5: 成功・失敗・却下・再実行・人承認の履歴が残らないと、後からなぜ通知・起票・修繕判断が行われたか説明できない。

## 5. AI解決
- C1対応:
  - OSS: n8n Agent nodeを、通知・起票を直接実行する本体ではなく、要約、分類、次アクション案の生成に限定する。実行ノードはSwitch、IF、Code、HTTP Requestなどで決定的に制御する。
  - 研究: OWASPの[Prompt Injection対策](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)では、最小権限、外部コンテンツの分離、人承認、攻撃シミュレーションが推奨されるため、AI Agentのツール権限を用途別に分ける。
  - 実運用: Redmine作成、Slack通知、メール送信、顧客連絡は「下書き作成」「社内通知」「承認後実行」の3段階に分ける。
- C2対応:
  - OSS: n8nのExecution data redaction、Credential管理、RBAC、Guardrailsを使い、APIキー、PII、顧客名、内部URLをログ・LLM入力・Slack本文から分離する。
  - 研究: OWASPのSensitive Information DisclosureとPrompt Injectionを参照し、外部テキスト、添付ファイル、Webページ、LLM応答を信頼済み命令として扱わない。
  - 実運用: LLMへ渡す点検情報は、チケットID、損傷種別、位置粒度、危険度、証跡URLなどに絞り、原画像や顧客詳細は権限付きリンクとして扱う。
- C3対応:
  - OSS: n8n Guardrails nodeで入力・出力を検査し、URL、PII、秘密鍵、Jailbreak、トピック逸脱をFail branchへ分岐する。
  - 研究: OWASPは期待出力形式の定義と決定的コードによる検証を推奨しているため、AI出力はJSON Schema相当の固定項目にしてから下流APIへ渡す。
  - 実運用: Redmine件名、カテゴリ、担当者、期限、優先度、説明、添付URL、承認者を必須項目化し、欠落時は起票せず人確認キューへ戻す。
- C4対応:
  - OSS: n8n Wait nodeのWebhook再開やフォーム提出再開を使い、危険度高、顧客通知、修繕判断、チケット終了は承認待ちにする。
  - 研究: OWASPのExcessive Agencyは、LLMに過度な自律実行権限を与えるリスクを扱うため、承認境界を「通知」「起票」「顧客連絡」「終了処理」に置く。
  - 実運用: AIは「推奨アクション」を出し、人間は「承認、差戻し、現地再確認、顧客連絡禁止」を選択する。承認者と判断理由をExecution DataとRedmineコメントに残す。
- C5対応:
  - OSS: Execution Data node、Redmine notes、Slack thread、n8n execution logを組み合わせ、イベントID、入力ソース、AI要約、承認者、実行結果を追跡する。
  - 研究: LLMハーネスの考え方を使い、AI出力の品質だけでなく、ツール実行、失敗分岐、承認結果、再実行率を評価する。
  - 実運用: `incident_id`、`source_event_id`、`workflow_execution_id`、`redmine_issue_id`、`slack_thread_ts`を共通キーにして、二重通知と二重起票を抑制する。

## 6. 価値
- C1由来: AI Agentを直接実行者ではなく判断補助に置くことで、n8nのAgent nodeが複数回ツールを選ぶ性質を活かしつつ、RedmineやSlackへの副作用をワークフロー側で制御できる。
- C2由来: n8nの実行データ秘匿は、本番実行の監視を残しながら機密ペイロードを隠せるため、点検画像、顧客情報、APIトークンを扱うワークフローの監査性と秘匿性を両立しやすい。
- C3由来: Guardrailsと固定出力形式を併用すると、AI出力がRedmine APIやSlack通知へ入る前に、PII、秘密鍵、URL、Jailbreak、形式不備を止められる。
- C4由来: Wait nodeによる承認待ちを入れると、危険度高の異常、補修判断、顧客通知、チケット終了を人が止められる。これはOWASPのExcessive Agency対策と整合する。
- C5由来: Execution Data、Redmine notes、Slack threadを共通IDで接続すると、通知、起票、承認、差戻し、完了の説明責任を後から確認できる。

## 7. トピック一覧
### n8n AI Agent
- 概要: [n8n Agentの説明](https://docs.n8n.io/advanced-ai/examples/understand-agents/)では、Agentは問い合わせに応じてツールを選び、ツール応答を評価して応答を作る構成として説明される。
- 特徴:
  - 複数回の実行でツール呼び出しと応答評価を行う。
  - 点検業務では、要約、分類、推奨アクション案の生成に向く。
  - 副作用のあるツール実行はワークフロー側で制限する必要がある。
- 主要スペック or 数値: Agent nodeは設定により異なるAgent種別として動作する。
- 制約 or 注意点: AIにツール選択を任せるほど、権限設計と承認境界が重要になる。

### Human-in-the-loop
- 概要: [n8n Human-in-the-loop for tool calls](https://docs.n8n.io/advanced-ai/human-in-the-loop-tools/)は、AIツール呼び出しに人確認を入れる設計の参照先である。
- 特徴:
  - 危険なツール呼び出しを人が承認・却下できる。
  - 点検・異常対応では、顧客通知、Redmine終了、修繕判断に適用しやすい。
  - 承認理由をログに残すことで監査性を高められる。
- 主要スペック or 数値: 不明。
- 制約 or 注意点: すべてを承認待ちにすると運用が遅くなるため、危険度別に承認要否を分ける。

### Wait nodeによる承認待ち
- 概要: [n8n Wait node](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.wait/)は、一定時間、指定時刻、Webhook、フォーム提出でワークフローを再開できる。
- 特徴:
  - 現地確認待ち、上長承認待ち、顧客連絡前確認に使える。
  - Limit Wait Timeを使うと、承認がない場合のタイムアウト分岐を設計できる。
  - フォーム提出を承認UIとして使える。
- 主要スペック or 数値: Resume条件はAfter Time Interval、At Specified Time、On Webhook Call、On Form Submitted。
- 制約 or 注意点: 長時間待機時は実行データ保存、再開URL、期限切れ処理を設計する。

### Guardrails node
- 概要: [n8n Guardrails node](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-langchain.guardrails/)は、AIモデルへ送る前の入力やAI出力を安全・セキュリティ・コンテンツポリシーで検査できる。
- 特徴:
  - Jailbreak、PII、Secret Keys、URL、Topical Alignmentなどを検査できる。
  - 違反時はFail branchへ分岐できる。
  - AI出力をRedmineやSlackへ渡す前の検査に使える。
- 主要スペック or 数値: JailbreakやTopical Alignmentは0.0から1.0のThresholdを設定できる。
- 制約 or 注意点: Guardrailsだけで安全が保証されるわけではなく、固定形式検証と人承認を併用する。

### Execution data redaction
- 概要: [Execution data redaction](https://docs.n8n.io/workflows/executions/execution-data-redaction/)は、ワークフロー実行データを秘匿しながら実行監視を残すための機能である。
- 特徴:
  - 本番実行データと手動実行データを別々に秘匿できる。
  - Instance owners/adminsがインスタンス全体に最低秘匿ポリシーを強制できる。
  - PII、金融情報、認証トークンを扱うワークフローで有効である。
- 主要スペック or 数値: 本番実行のみ、または手動・本番両方の秘匿を選べる。
- 制約 or 注意点: デバッグ性と秘匿性のバランスを決め、テスト環境に本番相当データを使う場合は手動実行も秘匿する。

### Execution Data node
- 概要: [Execution Data node](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.executiondata/)は、ワークフロー実行に検索可能なメタデータを保存する。
- 特徴:
  - 実行一覧で検索できるメタデータを残せる。
  - incident_id、source_event_id、redmine_issue_idなどの追跡キーに向く。
  - Redmine notesやSlack threadと組み合わせると追跡しやすい。
- 主要スペック or 数値: keyは50文字、valueは512文字までで、超過時は切り詰められる。
- 制約 or 注意点: 長文の点検内容や画像情報はメタデータではなく、権限付き保管先URLやチケット本文に分ける。

### Redmine連携
- 概要: [Redmine REST API](https://www.redmine.org/projects/redmine/wiki/Rest_api)はJSON/XMLでIssueなどのリソースを扱い、[Issues API](https://www.redmine.org/projects/redmine/wiki/Rest_Issues)はIssue作成・更新・コメント・添付を扱う。
- 特徴:
  - Issue作成はproject_id、subject、priority_id等を受け取る。
  - 更新時はnotesを付けられる。
  - 添付ファイルはupload tokenを使ってIssueへ紐付けられる。
- 主要スペック or 数値: Redmine APIのIssuesはStableでAvailability 1.0。
- 制約 or 注意点: AIから直接DELETEや終了ステータス更新を行わせず、担当者・期限・カテゴリ・説明・証跡を検証してから実行する。

### Slack通知
- 概要: [Slack Incoming Webhooks](https://docs.slack.dev/messaging/sending-messages-using-incoming-webhooks/)はSlackへメッセージを投稿する仕組みで、点検異常の社内通知に使える。
- 特徴:
  - Webhook URLへJSON payloadをPOSTして通知できる。
  - スレッド返信にはthread_tsが必要である。
  - Incoming Webhooksでは既定チャンネル、ユーザー名、アイコンはアプリ設定を継承する。
- 主要スペック or 数値: 不明。
- 制約 or 注意点: Webhook URLは秘密情報として扱い、顧客名や詳細画像をチャンネル本文へ直接出さず、権限付きリンクにする。

- 比較表:

  |---|---|---|---|---|---|---|---|
  | n8n AI Agent | ツール選択と応答生成を担う | モデル・設計依存 | LLM費用が発生 | 中 | 高 | 複数回実行でツール応答を評価 | [根拠](https://docs.n8n.io/advanced-ai/examples/understand-agents/) |
  | Human-in-the-loop | 危険なツール実行に人承認を入れる | 人判断依存 | 承認工数あり | 中 | 高 | 不明 | [根拠](https://docs.n8n.io/advanced-ai/human-in-the-loop-tools/) |
  | Wait node | 承認待ち・外部イベント待ちを実装 | 高 | 低 | 低〜中 | 高 | 4種類のResume条件 | [根拠](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.wait/) |
  | Guardrails node | 入力・出力の安全検査 | 閾値・モデル依存 | LLM利用時は費用あり | 中 | 中 | Threshold 0.0-1.0 | [根拠](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-langchain.guardrails/) |
  | Execution data redaction | 実行データ秘匿 | 高 | プラン・運用次第 | 中 | 中 | 本番/手動実行の秘匿設定 | [根拠](https://docs.n8n.io/workflows/executions/execution-data-redaction/) |
  | Execution Data node | 実行メタデータ検索 | 高 | 低 | 低 | 中 | key 50文字、value 512文字 | [根拠](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.executiondata/) |
  | Redmine REST API | 起票・更新・コメント | 高 | 既存Redmine費用 | 中 | 高 | Issues APIはStable | [根拠](https://www.redmine.org/projects/redmine/wiki/Rest_api) |
  | Slack Incoming Webhooks | 社内通知 | 高 | Slackプラン次第 | 低 | 中 | thread_tsでスレッド返信 | [根拠](https://docs.slack.dev/messaging/sending-messages-using-incoming-webhooks/) |
- 違い: Agentは判断補助、Wait/Human-in-the-loopは承認、Guardrailsは入出力検査、Execution Dataは監査、Redmine/Slackは業務接続を担う。
- 使い分け: AIが作るのは要約・分類・推奨案までにし、通知・起票・終了・顧客連絡は固定ロジックと承認を通して実行する。
- 強み / 弱み: n8nはAI、承認、API連携、ログを同じワークフローに置けるのが強みだが、設計を誤るとLLMの過剰な代理実行が業務システムへ直結する。
