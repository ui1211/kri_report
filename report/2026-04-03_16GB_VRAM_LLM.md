# 16GB VRAM 向けローカル LLM モデル

## 1. メタ情報

- 作成日時：2026-04-11
- 入力概要：既存レポート更新（最新情報重視）。既存候補に加えて「そのほか有効な LLM」を深掘りし、16GB VRAM 実運用に寄せて再整理。
- 更新方法：最新情報で更新
- トピック数：10
- 更新日：2026-04-25

## 2. 概要

16GB VRAM 環境では「完全 GPU 収容モデル」と「CPU オフロード前提モデル」で体感性能が大きく分かれるため、実運用では速度・品質・VRAM 収容性の三点最適化が重要である。実測系ベンチマーク（[Glukhov](https://www.glukhov.org/llm-performance/benchmarks/choosing-best-llm-for-ollama-on-16gb-vram-gpu/)）と配布実態（[Ollama Library](https://ollama.com/library?sort=popular)）を基準に、現状→課題→AI 解決→価値の順で整理した。2026 年 4 月時点では、Gemma 4 シリーズや Llama 4 が追加され、MoE アーキテクチャが主流化している。コーディング分野ではオープンソースモデルがクローズドモデルの SWE-bench スコアを凌駕する状況となっている。

## 3. 現状

- 業界・領域全体の状況
  - 16GB VRAM では、Q4 量子化で 8B〜20B 級の一部が「100% GPU 収容」で実用速度を出しやすい一方、24B+ は CPU オフロードが増えやすい（[Glukhov 実測](https://www.glukhov.org/llm-performance/benchmarks/choosing-best-llm-for-ollama-on-16gb-vram-gpu/)）。
  - 配布面では、`llama3.1` / `deepseek-r1` / `qwen2.5-coder` / `gpt-oss` などがローカル実行の主要選択肢として定着（[Ollama Library](https://ollama.com/library?sort=popular)）。
  - Cline は Plan/Act モードでモデルを切替可能であり、Plan は長文脈・推論、Act は速度・コード修正能力が実運用上の分岐点となる（[Cline Plan & Act](https://docs.cline.bot/core-workflows/plan-and-act)）。
  - 2026 年 4 月時点では、Gemma 4 シリーズや Llama 4 が追加され、MoE アーキテクチャが主流化。コーディング分野では MiniMax M2.5（SWE-bench 80.2%）や Kimi K2.5（76.8%）がクローズドモデルを凌駕。

- 実運用事例（リンク付き）
  - RTX 4080 16GB + Ollama 0.17.7 で、`gpt-oss:20b` 139.93 tok/s、`qwen3.5:9b` 90.89 tok/s、`ministral-3:14b` 70.13 tok/s の実測が報告されている（[Glukhov ベンチマーク表](https://www.glukhov.org/llm-performance/benchmarks/choosing-best-llm-for-ollama-on-16gb-vram-gpu/)）。
  - `deepseek-r1:14b` は Ollama 上で Q4_K_M 9.0GB（14.8B）として配布され、16GB クラスでの推論系用途の候補になっている（[Ollama deepseek-r1:14b](https://ollama.com/library/deepseek-r1:14b)）。
  - Gemma 4 26B MoE は 16GB VRAM で完全 GPU 収容（Q4_K_M）可能で、70 tok/s 前後の実測が期待される。
  - Llama 4 Scout（109B/Active 17B）は 61GB VRAM を要求するため、16GB 環境では CPU オフロードが必須。

## 4. 課題

- C1: VRAM 制約とオフロード劣化
  - 16GB を超えると CPU/GPU 分割が増え、応答速度が急低下しやすい。
- C2: Plan/Act で必要能力が異なる
  - Plan は長文脈・推論、Act はコード生成/修正と低遅延が求められ、単一モデル最適化が難しい。
- C3: コーディング精度の評価軸が分散
  - McEval / EvalPlus / LiveCodeBench / Aider など複数軸を跨いだ判断が必要。
- C4: 日本語実務での品質差
  - 英語系ベンチ強者が日本語実務で必ずしも最適とは限らない。
- C5: モデル更新が速く、陳腐化しやすい
  - 数か月単位で新系列・新蒸留版が出るため、比較表の鮮度維持が難しい。

## 5. AI 解決

- C1 対応:
  - OSS: Q4_K_M を基準に 100% GPU 収容モデルを優先（[Ollama model tags](https://ollama.com/library?sort=popular)）。
  - 研究：小型化・蒸留・量子化で推論効率を上げるアプローチ。
  - 実運用：実測で高速な `gpt-oss:20b` / `qwen3.5:9b` を優先し、24B+ は用途限定で採用（[Glukhov](https://www.glukhov.org/llm-performance/benchmarks/choosing-best-llm-for-ollama-on-16gb-vram-gpu/)）。
- C2 対応:
  - OSS: Cline の Plan/Act 切替を前提にモデルを分離運用（[Cline Plan & Act](https://docs.cline.bot/core-workflows/plan-and-act)）。
  - 研究：長文脈維持と推論品質を両立する運用設計（要約・分割・再投入）。
  - 実運用：Plan=長文脈寄り（Qwen3.5-9B など）、Act=コード特化（Qwen2.5-Coder / DeepSeek-R1 Distill）。
- C3 対応:
  - OSS: コード特化モデル（Qwen2.5-Coder, DeepSeek-R1 Distill）を明示的に採用。
  - 研究：複数ベンチの複合評価で過学習/偏りを緩和。
  - 実運用：Qwen2.5-Coder は公式公開で McEval 65.9、Aider 73.7 が示される（[Qwen2.5-Coder 公式](https://www.alibabacloud.com/blog/601765)）。
- C4 対応:
  - OSS: 日本語比率の高いモデル（LLM-jp-4, GPT-OSS Swallow）を選択肢に組み込む。
  - 研究：日本語評価データ拡張とタスク別再学習。
  - 実運用：日本語用途は汎用 SOTA より日本語特化モデルを優先配置（[NII LLM-jp-4](https://www.nii.ac.jp/news/release/2026/0403.html), [Swallow](https://swallow-llm.github.io/gptoss-swallow.ja.html)）。
- C5 対応:
  - OSS: Ollama タグ更新・モデル差し替えを前提に運用。
  - 研究：継続ベンチと差分追跡の自動化。
  - 実運用：月次で「速度/VRAM/文脈/コード精度」を再採点し、推奨 Tier を再編。

## 6. 価値

- コスト削減：クラウド API 常時利用からローカル実行へ寄せることで従量課金を圧縮できる。
- 性能最適化：16GB で完全 GPU 収容できるモデルは、重オフロード構成より大幅に高速（例：139.93 tok/s vs 12.64 tok/s、[Glukhov](https://www.glukhov.org/llm-performance/benchmarks/choosing-best-llm-for-ollama-on-16gb-vram-gpu/)）。
- 生産性向上：Plan/Act 分離で「設計品質」と「実装速度」を同時に最適化しやすい（[Cline Plan & Act](https://docs.cline.bot/core-workflows/plan-and-act)）。
- リスク低減：日本語タスクを特化モデルで補完することで、実務での誤解釈リスクを低減。
- コーディング性能：SWE-bench ベンチマークで、MiniMax M2.5（80.2%）や Kimi K2.5（76.8%）が Claude Opus 4.6 に肉薄し、クローズドモデルのリードを解消している。

## 8. 比較まとめ

- 比較軸:
  - 精度
  - コスト
  - 導入難易度
  - 拡張性

- 違い:
  - **速度最優先**: GPT-OSS-20B（139.93 tok/s）
  - **総合バランス**: Qwen3.5-9B（90.89 tok/s）
  - **コード特化**: MiniMax M2.5 / Kimi K2.5 / Qwen2.5-Coder-14B / DeepSeek-R1-Distill-Qwen-14B
  - **長文脈重視**: Gemma 4 31B / Mistral NeMo 12B
  - **日本語重視**: Qwen3.5 シリーズ / LLM-jp-4-8B / Gemma 4
  - **MoE 効率**: Llama 4 Scout / Gemma 4 26B MoE / Qwen3.5-397B
  - **大規模コーディング**: GLM-5.1 / MiniMax M2.5 / Kimi K2.5

- 使い分け:
  - Plan モード：Qwen3.5-9B, Gemma 4 31B, Mistral NeMo 12B
  - Act モード：Qwen2.5-Coder-14B, DeepSeek-R1-Distill-Qwen-14B, MiniMax M2.5
  - 軽量汎用：Llama 3.1 8B, Phi-4-mini-instruct
  - 日本語優先：Qwen3.5-9B, Gemma 4, LLM-jp-4-8B
  - 大規模環境：GLM-5.1, Kimi K2.5, MiniMax M2.5, Llama 4 Scout

- 強み / 弱み:
  - GPT-OSS-20B: 強み=速度、弱み=用途次第で文脈・日本語最適化が不足
  - Qwen3.5-9B: 強み=バランス、弱み=専門コードでは専用モデルに劣る場合
  - Qwen2.5-Coder-14B: 強み=コード、弱み=汎用会話では過剰最適化の可能性
  - MiniMax M2.5: 強み=SWE-bench 80.2%、弱み=Modified MIT 条項と大規模 VRAM 要求
  - Kimi K2.5: 強み=SWE-bench 76.8%、弱み=MIT ライセンスでも商用条項確認
  - GLM-5.1: 強み=SWE-bench 58.4%、弱み=日本語非最適化
  - Gemma 4 26B MoE: 強み=MoE 効率性、弱み=16GB では文脈制限
  - Llama 4 Scout: 強み=Meta 公式 MoE、弱み=Meta License 条項と 61GB VRAM 要求
  - Mistral NeMo 12B: 強み=128k 文脈、弱み=16GB 上での実効速度は要検証
  - DeepSeek-R1-Distill-14B: 強み=推論、弱み=対話スタイルが用途を選ぶ
  - LLM-jp-4-8B: 強み=日本語特化、弱み=詳細ベンチデータ不足
  - GPT-OSS Swallow: 強み=日本語推理、弱み=国際ベンチ比較不足
