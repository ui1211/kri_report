## 概要

ハーネスエンジニアリング（Harness Engineering）は、AI エージェント（特にコーディングエージェント）が信頼性高く機能するよう、モデル自体以外を構成するシステムを設計する工程である [[R-001](#R-001), [R-002](#R-002), [R-003](#R-003)]。OpenAI が 2026 年初頭に提唱した概念で、人間はコード実装から環境設計、意図の明確化、構造化されたフィードバックの提供へと焦点を移行することになる [[R-001](#R-001), [R-004](#R-004)]。

ステアリングループ（Steering Loop）は、ハーネスエンジニアリングの中核となる概念であり、人間がハーネスを反復して改善しながらエージェントを導く制御プロセスを表す [[R-001](#R-001), [R-003](#R-003)]。

### Agent = Model + Harness

この公式により、モデルがインテリジェンスと推論を提供する一方で、ハーネスが論理、ツール、安全性を提供する役割を明確にする [[R-002](#R-002), [R-004](#R-004)]。モデル alone ではデモに過ぎず、ハーネスによってプロダクト化される [[R-002](#R-002)]。

### Meta-Harness: ハーネスエンジニアリングの自動化

Stanford AI Lab の研究チームが発表した **Meta-Harness** は、ハーネスエンジニアリング自体を自動化するシステムである [[R-005](#R-005), [R-006](#R-006)]。これにより、人間が手作業でハーネスを最適化する必要がなくなり、コーディングエージェント自身がより良いハーネスを自動生成する反復ループを回すことができる。

**主要成果:**
- オンラインテキスト分類: 既存ハーネスより 7.7 ポイント高い精度 [[R-005](#R-005)]
- 数学推論: IMO レベル問題で 4.7 ポイントの向上 [[R-005](#R-005)]
- 自動コーディング: 手動設計ハーネスを上回るパフォーマンス [[R-005](#R-005)]

## 特徴

### 1. Feedforward and Feedback Controls

ハーネスは二種類の制御を組み合わせてエージェントを制御する：

#### Feedforward Controls（事前制御）
- エージェントが行動する **前に** 意図する出力を anticipates し、問題発生を防ぐ
- 確率的に良い結果を創出する
- **例**: システムプロンプト、ツール/スキル/MCP の説明、バンドルされたインフラストラクチャ、オアケストレーションロジック、フック/ミドルウェア [[R-002](#R-002)]

#### Feedback Controls（事後制御）
- エージェントが行動した **後に** 観測され、自己修正を助ける
- LLM に最適化された信号（例：指示を含むカスタムリンターメッセージ）が特に強力
- **例**: コードレビューエージェント、静的解析（linters）、ログ監視、ブラウザ検証、CI/CD パイプライン [[R-001](#R-001), [R-003](#R-003)]

> **重要な知見**: Feedforward-only や feedback-only のどちらか一方だけでは、同じ過ちを繰り返すエージェントか、機能しないエージェントになる [[R-001](#R-001)]

### 2. Computational vs Inferential Controls

| 制御タイプ | 実行方法 | 特徴 | 例 |
|------------|----------|------|-----|
| **Computational** | CPU | 決定論的、高速、信頼性あり | テスト、リンター、型チェッカー、静的解析 [[R-001](#R-001)] |
| **Inferential** | GPU/NPU | 意味的解析、非決定論的、高コスト | LLM as judge、AI コードレビュー [[R-001](#R-001)] |

Computational controls は毎コミットに実行可能で低コスト、Inferential controls は高価だが豊かなガイダンスを提供 [[R-001](#R-001)]。

### 3. Regulation Categories

ハーネスはコードベースを所望の状態へ制御するサイバネティックなガバナ役を担う [[R-001](#R-001)]。

#### Maintainability Harness（保守性ハーネス）
- コード品質と保守性を制御
- Computational sensors で構造的問題（重複コード、シクロマティック複雑度、テストレース欠如、アーキテクチャドリフト、スタイル違反）を確実に検出
- Inferential controls は意味的に重複したコードや冗長なテストを部分的に扱う

#### Architecture Fitness Harness（アーキテクチャ適合性ハーネス）
- アプリケーションのアーキテクチャ特性を定義・検証
- 例：パフォーマンス要件を feedforward、パフォーマンステストを feedback [[R-001](#R-001)]
- 可観測性のコーディング慣習やデバッグ指示 [[R-001](#R-001)]

#### Behaviour Harness（行動ハーネス）
- アプリケーションが機能的行为するかを制御
- Feed-forward: 機能的な仕様にし、Feedback: AI 生成のテストスイートがグリーンかをチェック
- 機能的行为ハーネスにはまだ開発余地がある [[R-001](#R-001)]

### 4. Core Harness Components

LangChain の分析に基づく主要構成要素 [[R-002](#R-002)]：

1. **System Prompts**: エージェントの行動を定義する初期指示
2. **Tools, Skills, MCPs**: エージェントが利用可能な機能
3. **Bundled Infrastructure**: ファイルシステム、サンドボックス、ブラウザ
4. **Orchestration Logic**: サブエージェントの生成、ハンドオフ、モデルルーティング
5. **Hooks/Middleware**: 決定論的実行のためのチェック

### 5. Filesystems and Durable Storage

ファイルシステムは最も基本的なハーネスプリミティブ：
- エージェントに作業スペースを提供
- 情報オフロード（コンテキストウィンドウを超える）
- セッション間での状態保持
- 複数エージェント/人間間の協調表面 [[R-002](#R-002)]

Git はバージョン管理、ロールバック、実験分岐を可能にする [[R-002](#R-002)]。

### 6. Sandboxes and Safety

- ローカル実行のリスクを sandbox で軽減
- コマンドの許可リスト、ネットワーク隔離でセキュリティ
- 必要環境のオンデマンド構築、並列化、片付け [[R-002](#R-002)]

### 7. Memory & Search

- **ファイルシステム**: AGENTS.md などの標準化された記憶形式
- **継続学習**: セッション間の知識保存と注釈付け
- **Web Search/MCP**: トレーニングのカットオフを超える最新情報のアクセス [[R-002](#R-002)]

### 8. Context Management Strategies

#### Compaction
- コンテキストウィンドウが充填された時の処理
- 既存コンテキストをオフロード・要約 [[R-002](#R-002)]

#### Tool Call Offloading
- 大規模ツール出力のノイズ削減
- ヘッダー/フッタートークンの保持、完全出力のファイルシステムへのオフロード [[R-002](#R-002)]

#### Skills（段階的開示）
- コンテキストウィンドウ充填前のパフォーマンス低下防止
- 開始時のスキルフロントマターの不要化 [[R-002](#R-002)]

### 9. Meta-Harness の特徴

Meta-Harness はハーネスエンジニアリングを自動化する outer-loop システムである [[R-005](#R-005), [R-006](#R-006)]。

####  proposer（提案エージェント）
- コーディングエージェントを使用してハーネスを提案
- ファイルシステムを通じて全ての prior experience にアクセス可能
- 既存ハーネスのソースコード、実行トレース、スコアを選択的に閲覧

#### Filesystem Access
- 1 回の反復で平均 82 ファイルを閲覧 [[R-005](#R-005)]
- prior harnesses、scores、execution traces にアクセス
- シンプルな grep/cat コマンドで情報を取得

#### Search Loop
1. **提案**: prior experience を閲覧して新しいハーネスを提案
2. **評価**: 提案ハーネスを検索セットで評価
3. **記録**: コード、スコア、実行トレースをファイルシステムに記録
4. **反復**: ループを継続 [[R-005](#R-005)]

#### 非圧縮されたフィードバック
- 従来の text optimizer はフィードバックを過度に圧縮（100〜30,000 tokens）
- Meta-Harness は 1 回の評価で最大 10,000,000 tokens の診断情報を提供 [[R-005](#R-005)]
- 完全な実行履歴を保持することで、因果関係の診断を可能化

## 制約 / 注意点

### 1. Harnessability の限界

すべてのコードベースが同等にハーネス可能ではない [[R-001](#R-001)]：

- **強 typed ランゲージ**: 型チェッキングセンサーが利用可能
- **明確なモジュール境界**: 制約ルールが容易
- **フレームワーク（Spring など）**: 明示的でない制御を内包

動的タイピング言語はハーネス構築が困難 [[R-001](#R-001)]。

### 2. Greenfield vs Legacy

#### Greenfield チーム
- 最初からハーネス性を埋め込む
- 技術決定とアーキテクチャ選択で制御可能度を決定 [[R-001](#R-001)]

#### Legacy チーム
- ハーネスが最も必要だが構築が困難
- 技術的負債の蓄積で制御が複雑 [[R-001](#R-001)]

### 3. Harness Templates の課題

- バージョン管理と同期の問題
- 非決定論的ガイド/センサーはテスト困難
- アップストリーム改善との同期困難 [[R-001](#R-001)]

### 4. Co-evolution の副作用

モデルとハーネスの共進化：
- ツールロジック変更はモデルパフォーマンス低下
- ハーネス過適合（過学習）[[R-002](#R-002)]
- テーミナルベンチなどで他のハーネスで低スコア

### 5. Behavioral Harness の不足

- 機能的行为ハーネスにはまだ発展余地
- AI 生成のテストへの過信は不十分 [[R-001](#R-001)]
- Approved fixtures パターンは限定的に適用

### 6. Context Rot

エージェントのパフォーマンスが劣化する問題：
- コンテキストウィンドウ充填による推論低下
- Compaction、ツールオフロード、スキルで緩和 [[R-002](#R-002)]

### 7. Harness Coverage 測定の未解決

- コードカバレッジやミューテーションテストのような
  ハーネスカバレッジ測定方法の開発必要 [[R-001](#R-001)]

### 8. Open Questions [[R-001](#R-001)]

- ハーネスの成長に伴う整合性の維持
- 矛盾する指示・フィードバック信号のトレードオフ
- センサー発火の不在が品質か不十分かの判断
- ハーネスカバレッジと品質の評価メトリクス

### 9. Meta-Harness の限界 [[R-005](#R-005)]

- **検索セット過適合**: テストセット結果を提案者は見ないが、検索セットに過適合する可能性
- **計算コスト**: 1 回の反復で数十のハーネス評価が必要
- **プロンプトエンジニアリング依存**: skill テキストの質が検索結果に大きく影響
- **ドメイン特異性**: 特定領域で最適化されたハーネスは他の領域で機能しない [[R-005](#R-005)]

### 10. Meta-Harness の実装課題

- **ファイルシステム管理**: 検索履歴が膨大な場合にナビゲーションが複雑
- **評価コスト**: 高速で判別可能な評価関数が必須
- **検索セット設計**: 既によくできてしまった検索セットでは最適化余地が限定的 [[R-005](#R-005)]

### 違い

#### ハーネスエンジニアリング vs プロンプトエンジニアリング

| 項目 | プロンプトエンジニアリング | ハーネスエンジニアリング |
|------|---------------------------|-------------------------|
| 焦点 | 指示の最適化 | 環境・制御システムの設計 |
| 信頼性 | 非決定論的、不安定 | 決定論的センサーによる安定性 |
| 持続性 | セッション限定 | セッション間継続学習 |
| 制御 | 自然言語指示 | コンピュータブル/インフェリンシャル制御 |
| 検証 | 主観的評価 | 構造テスト、CI/CD、センサー |

#### Model vs Harness

| 要素 | Model | Harness |
|------|-------|---------|
| 役割 | インテリジェンス、推論 | 論理、ツール、安全性 |
| 制御 | 自然言語理解 | 制御論的制御 |
| 持続性 | トレーニングカットオフ | ファイルシステム、メモリ |
| 実行 | 出力のみ | エグゼクション環境 |
| 検証 | 自己評価 | 構造テスト、フィードバックループ |

#### Meta-Harness vs 従来手法

| 項目 | 従来手法 | Meta-Harness |
|------|--------|--------------|
| 最適化対象 | プロンプトのみ | ハーネス全体 |
| 履歴利用 | 圧縮済み（100〜30K tokens） | 完全履歴（最大 10M tokens） |
| 検索速度 | 数千〜数万評価 | 数百評価 |
| 一般化 | 限定的 | OOD データセットで優位 [[R-005](#R-005)] |
| 自動化 | 人手作業 | 自動化 [[R-005](#R-005)] |

### 強み / 弱み

#### ハーネスエンジニアリング

**強み**:
- エージェント信頼性の向上 [[R-002](#R-002), [R-003](#R-003)]
- セッション間での状態保持
- 自動化された自己修正
- 複数エージェント間の協調
- コストと速度のトレードオフ管理 [[R-001](#R-001)]

**弱み**:
- 設計コストが高い [[R-001](#R-001)]
- 非決定論的センサーのテスト困難
- 既存コードベースへの適用困難
- Harnessability 不均一 [[R-001](#R-001)]

#### ステアリングループ

**強み**:
- 人間による継続的な改善
- Feedforward/feedbackの両方活用
- ハーネスの自己最適化
- 問題の事前予防 [[R-001](#R-001)]

**弱み**:
- 人間の介入が必要
- 経験に依存
- 反復の時間コスト [[R-001](#R-001)]

#### Codex

**強み**:
- IDE 内での直接連携
- モデル切り替え可能
- クラウド委任
- 60k stars のコミュニティ [[R-004](#R-004)]

**弱み**:
- Windows 対応は実験的
- API キーまたは ChatGPT プラン必要
- Cloud 委任の監視必要 [[R-004](#R-004)]

#### Cline

**強み**:
- 完全な自律性と人間在ループ
- MCP でツール拡張
- ブラウザ機能
- チェックポイントとロールバック [[R-004](#R-004)]

**弱み**:
- 設定の複雑さ
- API プロバイダの選択必要
- 多言語サポート未整備 [[R-004](#R-004)]

#### Meta-Harness

**強み**:
- ハーネスエンジニアリングの自動化 [[R-005](#R-005)]

- 10 倍高速な最適化 [[R-005](#R-005)]
- 完全な実行履歴による診断 [[R-005](#R-005)]

**弱み**:

- ファイルシステム管理の複雑さ [[R-005](#R-005)]
