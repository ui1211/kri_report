## 2. 概要
- n8n は [公式サイト](https://n8n.io/)で提供されるワークフロー自動化プラットフォームで、ノードベースのビジュアルエディタ、セルフホスティング、クラウド利用、AI ノード、外部サービス連携を特徴とする。
- 現状では、Zapier や Make と比較される自動化ツールの一つであり、技術チームやセルフホスティング需要がある組織では、データ管理とコード拡張の柔軟性が価値になる。
- 課題は、学習コスト、セルフホスティング運用、Webhook や外部公開設定、Fair-code ライセンス理解、AI ワークフローの設計難度である。
- AI ノード、LangChain nodes、Chat Trigger、AI Transform、公式学習リソース、エラーハンドリング設計を組み合わせることで、自動分類、RAG チャットボット、データ分析レポートなどの業務自動化に展開できる。

## 3. 現状
- 業界・領域全体の状況: 業務自動化では、ノーコード / ローコードで SaaS、API、データベース、チャット、メールを接続する需要がある。n8n は [公式ドキュメント](https://docs.n8n.io/)で、ノード、ワークフロー、トリガー、データ変換、エラー処理、AI 連携などを提供している。
- 実運用事例: n8n はセルフホストと [n8n Cloud](https://docs.n8n.io/manage-cloud/overview/)の両方で導入できる。セルフホストでは Docker や npm による起動、クラウドでは AI Assistant、Concurrency、Source Control、Log Streaming、Security Settings などの管理機能が利用対象になる。
- 技術的特徴: ワークフローはノードと接続で構成され、JSON 形式のアイテムがフロー内を流れる。Core nodes、Built-in nodes、LangChain nodes、HTTP nodes、Code node を組み合わせることで、標準連携とカスタム処理を同じワークフロー内で扱える。
- 比較状況: 既存資料では、n8n は 400 以上の連携、Zapier は 8,000 以上のアプリ連携、Make は 1,000 以上の連携と整理されている。ただし、当該数値は本ファイル内の入力情報に基づくもので、最新値の再検証は未実施である。

## 4. 課題
- C1: n8n はノード、アイテム、データマッピング、分岐、マージ、ループ、エラー処理を理解する必要があり、単純なノーコードツールより学習コストが高くなりやすい。
- C2: セルフホスティングでは Docker 環境、データ永続化、バックアップ、SSL、Webhook の外部公開、権限管理を運用する必要がある。
- C3: AI ワークフローでは、AI ノードの出力品質、RAG 構成、外部 LLM 連携、個人情報や機密データの扱い、エラー時のフォールバックを設計しないと業務利用が不安定になりやすい。

## 5. AI解決
- C1対応:
  - OSS: [GitHub リポジトリ](https://github.com/n8n-io/n8n)で公開されている n8n 本体、公式ドキュメント、コミュニティ情報を参照し、ノード構造、データ構造、エラー処理を段階的に学ぶ。
  - 研究: 公式の Level 1 / Level 2 テキストコース、動画チュートリアル、[Advanced AI](https://docs.n8n.io/advanced-ai/intro-tutorial/)を使い、基本操作から AI ワークフローまで段階的に習得する。
  - 実運用: 小さなワークフローから始め、Webhook Trigger、HTTP Request、Code node、Slack / メール通知などの基本パターンをテンプレート化する。
- C2対応:
  - OSS: Docker によるセルフホスト、npm による開発環境起動、コミュニティテンプレートを活用して、インフラ構成を標準化する。
  - 研究: [セルフホスティング設定](https://docs.n8n.io/hosting/)、[Error handling](https://docs.n8n.io/flow-logic/error-handling/)、[Concurrency](https://docs.n8n.io/manage-cloud/concurrency/)を確認し、永続化、同時実行、失敗時処理を設計する。
  - 実運用: 本番ではデータ永続化、バックアップ、SSL、Webhook URL、ログ監視、認証設定を明示し、ローカル検証では ngrok などのトンネリング利用を検討する。
- C3対応:
  - OSS: n8n の LangChain nodes、HTTP nodes、Code node を組み合わせ、LLM、検索、外部 API、通知を一つのワークフローに統合する。
  - 研究: [Advanced AI](https://docs.n8n.io/advanced-ai/)の学習を通じて、Chat Trigger、AI Transform、RAG、LLM 連携の構成を確認する。
  - 実運用: メール分類、RAG チャットボット、スプレッドシート分析などは、AI 出力の確認、失敗時分岐、ログ保存、機密データの扱いを含めて設計する。

## 6. 価値
- C1由来: 公式コース、ドキュメント、コミュニティを使って学習経路を固定すると、ノード、データ構造、エラー処理、AI 連携を段階的に理解できる。既存資料では n8n は 400 以上の連携と整理されており、幅広い業務接続に使える。
- C2由来: セルフホスティングにより、機密データを自社管理下で処理できる可能性がある。クラウド専用ツールと異なり、Docker や npm で自社環境に導入できる点が比較上の価値になる。
- C3由来: AI Transform、Chat Trigger、LangChain nodes を使うことで、メール分類、RAG チャットボット、データ分析レポートなどをワークフロー内に組み込める。既存資料にあった「70〜80% の工数削減」や「月額 $100〜500 削減」は根拠リンクが不足しているため、本版では定量効果として採用しない。

### 技術スタックとアーキテクチャ
- 概要: [公式ドキュメント](https://docs.n8n.io/)では、n8n のワークフローはノード、接続、トリガー、データ変換、エラー処理などで構成される。

### 導入方法
- 概要: n8n は [セルフホスティング設定](https://docs.n8n.io/hosting/)に従って Docker や npm で導入できる。
- 特徴:
  - Docker は本番寄りの導入に使いやすい。
  - npm は開発環境や検証用途に使いやすい。
  - n8n Cloud を使うとインフラ管理を任せられる。
- 主要スペック or 数値: 代表コマンドは `docker run ... n8nio/n8n`、または `npm install -g n8n` と `n8n start`。
- 制約 or 注意点: セルフホストではデータ永続化、バックアップ、SSL、Webhook URL、認証設定を別途管理する。

### n8n Cloud
- 概要: [n8n Cloud](https://docs.n8n.io/manage-cloud/overview/)は、n8n をクラウド環境で利用する選択肢である。
- 特徴:
  - AI Assistant、Concurrency、Data Management、Source Control、Log Streaming、Security Settings が利用対象になる。
  - インフラ運用負荷をセルフホストより下げやすい。
  - クラウド管理機能とスケーラビリティを利用できる。
- 主要スペック or 数値: 既存資料では Free trial の記載があるが、プラン詳細と価格は最新確認が必要。
- 制約 or 注意点: 自社管理要件が強い場合はセルフホストとの比較が必要になる。

### AI 機能
- 概要: n8n は [Advanced AI](https://docs.n8n.io/advanced-ai/)で AI ワークフロー構築の情報を提供している。
- 特徴:
  - AI Transform、Chat Trigger、LangChain nodes を利用できる。
  - OpenAI、DeepSeek、Google Gemini、Groq、Azure などの LLM 連携を扱える。
  - RAG、分類、分析、通知などの業務ワークフローに組み込める。
- 主要スペック or 数値: 入力資料上、メール分類、RAG チャットボット、スプレッドシート分析が例として挙げられている。
- 制約 or 注意点: AI 出力の品質、コスト、機密データ、失敗時処理はワークフロー設計で制御する必要がある。

### エラーハンドリングと運用

- 特徴:
  - ワークフロー全体のエラー処理を設計できる。
  - 個々のノードの失敗時対応を考慮できる。
  - ログ監視や通知と組み合わせて運用しやすくなる。
- 主要スペック or 数値: 不明。
- 制約 or 注意点: 外部 API 連携では、リトライ、重複実行、部分失敗、通知漏れを設計する必要がある。
