# ハーネスエンジニアリングとステアリングループ

## 要約
ハーネスエンジニアリングは、AI エージェントの品質をモデルの性能ではなく、制御環境の設計で安定化させる考え方である。特にステアリングループは、人間が feedforward と feedback を使い分けながらハーネスを反復改善する中心プロセスであり、近年は Meta-Harness によってその改善ループ自体の自動化も研究対象になっている。要するに、エージェントは「モデル単体」ではなく「モデル + ハーネス」で評価すべきという整理である。

## 現状
現場では、システムプロンプト、ツール群、サンドボックス、ファイルシステム、レビュー、テスト、CI などをまとめた制御層がエージェント品質を左右している。ハーネスは事前制御で逸脱を防ぎ、事後制御で誤りを修正する役割を持つ。さらにコンテキスト管理、永続ストレージ、メモリ検索、権限制御まで含めて設計する必要があり、単なるプロンプト設計よりはるかに広い。Meta-Harness はその外側でハーネスコード自体を探索、評価、改善する outer-loop として提案され、既存手法を上回る性能報告も出ている。

## 課題
ハーネスは効果が高い一方、設計コストが大きく、既存コードベースほど整備が難しい。feedforward と feedback のどちらかに偏ると、同じ失敗の反復か、行動不能な厳格制御に寄りやすい。さらに、動的言語や技術的負債の大きい環境では harnessability 自体が低く、構造テストやセンサーを埋め込みにくい。Meta-Harness も万能ではなく、検索セットへの過適合、評価コスト、履歴管理の複雑さ、ドメイン依存性といった制約を抱える。

## 解決
実務ではまず、Computational controls と Inferential controls を役割分担して配置するのが有効である。テスト、型検査、リンターは低コストで常時実行し、意味的評価や AI レビューは高価だが重要な箇所に限定する。次に、ファイルシステムと durable storage を前提に、ログ、評価結果、実行履歴を蓄積し、ステアリングループを改善可能な形にする。新規プロジェクトでは最初からハーネス性を埋め込み、既存プロジェクトでは保守性ハーネスから段階導入するのが現実的である。Meta-Harness や AutoHarness は、その改善作業を自動化する補助層として使う位置づけが妥当である。

## 価値
この考え方を採ると、AI エージェント開発は不安定な試行錯誤から、制御工学に近い反復改善プロセスへ移る。ハーネス整備により、品質保証、安全性、再現性、監査性をコードとして管理できるため、長時間タスクや複数エージェント協調に強くなる。ステアリングループを明示化すれば、人間の役割も「都度介入」から「制御設計と改善」に変わり、属人化を減らせる。さらに Meta-Harness の発展は、将来的にハーネス改善そのものを自動化し、エージェント開発の生産性を一段押し上げる可能性がある。
