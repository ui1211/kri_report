# 物体検知モデル可視化用 OSS ツール調査レポート

## 要約
物体検知モデルの可視化を OSS で行う場合、実際の選択肢は「アノテーション兼可視化基盤」「監視ダッシュボード」「ブラウザ内実行」「用途特化 UI」に分かれる。完全にブラウザだけで完結する選択肢は少なく、多くはサーバー起動や Python 環境を前提とする。そのため、目的が学習データ整備なのか、推論結果確認なのか、運用監視なのかでツール選定を分ける必要がある。

## 現状
アノテーションと予測可視化の主力は Label Studio と CVAT で、Web UI とチーム運用機能を備え、ML 補助ラベリングやモデル予測表示まで含めた実務利用が進んでいる。運用監視では Evidently AI が指標可視化やドリフト検知を担い、Foxglove はロボティクスやセンサーデータ連携を含む 2D/3D 可視化に強い。軽量な試作やプライバシー重視の用途では Transformers.js がブラウザ内推論を可能にし、NiceGUI や Dash は Python ベースで独自ダッシュボードを作る選択肢になっている。

## 課題
大きな制約は、見た目が Web UI でも多くのツールが裏側でサーバー起動や Python 環境を要求する点である。Label Studio や CVAT は高機能だが初期設定と運用知識が必要で、Evidently AI や Dash、NiceGUI は Python 依存が前提になる。Transformers.js は完全クライアントサイドの利点がある一方、性能とメモリはブラウザ制約を強く受ける。つまり、ノーコードに見えても運用現場では環境構築、人材要件、可視化対象データ形式の整理が避けられない。

## 解決
用途別に基盤を分けるのが現実的である。学習データ作成と予測レビューは Label Studio または CVAT、継続監視は Evidently AI、ロボティクス可視化は Foxglove、試作や端末内処理は Transformers.js、独自 UI は NiceGUI か Dash と切り分けると導入判断がしやすい。サーバー起動が必要な前提は隠さず、Docker 化やテンプレート構成で配布し、現場のセットアップ負担を減らすのが有効である。完全ブラウザネイティブを求める場合は機能を絞り、Transformers.js や軽量 HTML ベース可視化に限定した方がよい。

## 価値
適切なツール選定ができれば、物体検知の可視化を単なる開発者向けデバッグから、現場レビューや継続運用のインターフェースへ広げられる。OSS を使うことでライセンスコストを抑えつつ、ML 補助ラベリング、監視、自動連携、プライバシー保護を目的別に取り込める。特に Label Studio や CVAT は業務導入の基盤として強く、Transformers.js はローカル完結性で差別化できる。結果として、モデル精度の評価、改善サイクル、説明可能性の実務運用を低コストで整備しやすくなる。
