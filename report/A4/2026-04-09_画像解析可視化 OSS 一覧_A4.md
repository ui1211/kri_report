# 2026-04-09 画像解析可視化 OSS 一覧

## 要約
画像解析の可視化 OSS は、単一の最適解がある市場ではなく、アノテーション基盤、監視ダッシュボード、リアルタイム推論 UI、説明用可視化、グラフ描画ライブラリに分かれている。したがって、何を可視化したいのかを定義せずにツール比較をしても実務判断につながりにくい。本資料では、Web UI を持つ実務向け基盤と、軽量な可視化部品群を用途別に整理した点に価値がある。

## 現状
業務利用の中心は Label Studio と CVAT で、アノテーション、予測結果表示、チーム運用まで一体で扱える。監視用途では Evidently AI がドリフトや性能変化の可視化を担い、リアルタイムやクライアントサイド寄りでは Transformers.js や MediaPipe が候補になる。独自 UI を手早く組みたい場合は Streamlit、Dash、NiceGUI、Panel などの Python ベースフレームワークが選ばれやすい。さらに CNN Explainer や Transformer Explainer は教育・理解用途、OpenCV や Plotly、Matplotlib は部品レベルの可視化基盤として使われる。

## 課題
多くの OSS は表面的にはブラウザ操作中心でも、裏側では Python 環境、サーバー起動、Docker 配備、追加依存を要求する。Transformers.js のようなブラウザ完結型は扱いやすいが、性能とメモリの制約が強い。逆に高機能な Label Studio や CVAT は設定が重く、Streamlit や Dash は実装自由度と引き換えに設計責任が増える。つまり、可視化 OSS 選定の難しさは機能差よりも、導入運用コストと開発体制との相性にある。

## 解決
用途別にツール層を分けるのが有効である。データ整備と予測レビューは Label Studio か CVAT、監視は Evidently、リアルタイム表示は MediaPipe や Transformers.js、業務向けダッシュボードは Streamlit、Dash、NiceGUI、Panel に任せる。教育・説明用途だけなら Explainer 系を単独で使い、基盤用途に流用しない方が整理しやすい。導入時は「サーバー起動不要か」「Python 必須か」「チーム共有が必要か」「モデル実行も含むか」という4軸で切ると、比較表が実務判断に変わる。

## 価値
この整理により、画像解析の可視化を単なる見栄えの問題ではなく、開発、検証、運用、説明責任を支える基盤選定として扱える。OSS を組み合わせれば、ライセンス費用を抑えつつ、アノテーション、監視、共有、リアルタイム表示を目的別に最適化できる。特に Label Studio や CVAT は業務基盤、Streamlit や Dash は内製 UI、Transformers.js はローカル完結性で差別化できる。結果として、モデル改善サイクルを可視化中心に再設計しやすくなり、現場への説明力も高まる。
