# 2026-04-04 LLM ハーネス・メタハーネス

## 要約
LLM 活用の成否はモデル単体ではなく、その外側で行動を制御するハーネス設計に強く依存する。本資料は、LLM の出力とツール利用を管理する「ハーネス」、そのハーネス自体を自動改善する「メタハーネス」、そしてローカルで使える OSS フレームワーク群を整理したものである。要点は、エージェント品質の差はプロンプトの巧拙より、制御構造、評価ループ、安全策の設計差として現れやすい点にある。

## 現状
ハーネスは、ツール定義、サンドボックス、安全制約、会話ループ、履歴管理を組み合わせて LLM の行動を制御する実装層として認識されている。メタハーネスはその一段外側で、既存ハーネスコードを探索、評価、改良し、性能向上やトークン効率改善を狙う研究領域として立ち上がってきた。実装面では LangChain、LangGraph、CrewAI、AutoGen、LlamaIndex、Semantic Kernel などの OSS が豊富で、ローカル LLM や Ollama と接続しながら独自の制御系を組める環境が整っている。

## 課題
人間がハーネスを設計する限り、制御ロジックは経験依存になりやすく、保守コストも増えやすい。メタハーネスは有望だが、完全自動化には至っておらず、評価指標や改善探索の妥当性を人間が監督する必要がある。OSS フレームワーク群は選択肢が多い反面、相互互換性が低く、日本語支援や長期メンテナンスの質にも差がある。結果として、モデル選定よりも「どのフレームワークで、どの安全制約と評価ループを標準化するか」が運用の難所になる。

## 解決
実務では、まずハーネスを最低限の構成要素に分解し、ツール権限、履歴管理、評価観点、停止条件を明示して設計することが有効である。その上で LangGraph や CrewAI などの既存フレームワークを土台にし、独自実装は差分部分に限定する。メタハーネスの考え方は、ハーネスコードを固定資産と見なさず、実行ログとスコアを元に継続改善する仕組みとして取り込むと価値が高い。日本語用途や社内業務では、日本語モデル、テンプレート化されたワークフロー、評価データ整備を併用して、海外ベース設計との差を埋める必要がある。

## 価値
この整理により、LLM エージェント開発を「良いモデル選び」から「良い制御環境設計」へ視点転換できる。ハーネスを整備すると、安全性、再現性、監査性が上がり、属人的なプロンプト調整から脱却しやすい。メタハーネスの発想を取り入れれば、運用ログを活かして制御コード自体を改善対象にできる。さらにローカル OSS 基盤を使えば、クラウド依存や機密データ流出リスクを抑えつつ、チームごとの運用ルールに合わせたエージェント基盤を内製しやすくなる。
