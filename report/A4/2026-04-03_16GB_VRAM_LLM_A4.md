# 16GB VRAM 向けローカル LLM モデル

## 要約
16GB VRAM 環境では、単に大きいモデルを選ぶよりも「完全 GPU 収容できるか」「Plan と Act で役割分担できるか」を軸に候補を絞る方が実運用で成果が出やすい。実測ベンチマークと Ollama 配布状況を見ると、速度重視、コード特化、日本語重視、長文脈重視で最適解が分かれており、単一モデル万能主義ではなく用途別の Tier 設計が現実的である。2026 年 4 月時点では、Gemma 4 シリーズや Llama 4 が追加され、MoE アーキテクチャが主流化。コーディング分野では MiniMax M2.5（SWE-bench 80.2%）や Kimi K2.5（76.8%）がクローズドモデルを凌駕している。

## 現状
16GB では Q4 量子化を前提に 8B から 20B 級の一部が 100% GPU 収容で実用速度を出しやすく、24B 以上は CPU オフロードが増えて急激に遅くなりやすい。現場候補としては、速度面で `gpt-oss:20b`、総合バランスで `qwen3.5:9b`、コード用途で `qwen2.5-coder` や `deepseek-r1:14b`、長文脈では Mistral NeMo、日本語用途では LLM-jp-4 や GPT-OSS Swallow が整理対象になる。加えて Cline の Plan/Act モードが定着しつつあり、長文脈推論と低遅延実装をモデル分離して扱う運用が前提化している。2026 年 4 月時点では、Gemma 4 26B MoE（16GB VRAM で 100% GPU 収容可能）や Llama 4 Scout（61GB VRAM 必要）が追加され、MoE アーキテクチャが主流化している。

## 課題
最大の制約は VRAM 超過による性能劣化で、16GB を少しでも越えると CPU/GPU 分割が増え、体感性能が大きく崩れる。さらに Plan と Act で必要能力が異なるため、推論重視とコード修正重視を単一モデルに両立させにくい。評価指標も McEval、EvalPlus、LiveCodeBench、Aiderなどに分散し、どの数値で選ぶかが難しい。加えて、日本語実務での適性は英語系ベンチの強さと一致せず、モデル更新速度も速いため、比較表がすぐ陳腐化する。

## 解決
実装方針としては、まず Q4_K_M 前後で 100% GPU 収容できるモデルを優先し、16GB 環境での実測速度を基準に Tier を固定する。運用上は Plan 用に Qwen3.5 系や Mistral NeMo のような長文脈寄りを置き、Act 用には Qwen2.5-Coder や DeepSeek-R1 Distill のようなコード特化系を分離する。日本語業務には LLM-jp-4 や Swallow 系を補助候補として加え、英語ベンチ偏重を避ける。さらに月次で速度、VRAM 使用量、文脈長、コード精度を再採点し、Ollama タグ更新と差し替えを前提に継続運用する構成が妥当である。

## 価値
この整理により、クラウド API への常時依存を減らしつつ、16GB クラスの単一 GPU でも実務に耐えるローカル LLM 構成を組みやすくなる。速度最適化では完全 GPU 収容モデルが重オフロード構成より大幅に有利で、gpt-oss:20b（139.93 tok/s）が gpt-oss:120b（12.64 tok/s）の 11 倍の速度を出す。開発体験の差が明確に出る。Plan/Act 分離を前提にすると、設計品質と実装速度を同時に改善しやすい。さらに日本語特化モデルを補助採用することで、社内文書や日本語 QA での誤解釈リスクを下げられ、現場運用に耐える選定基準を持てる。コーディング性能においては、SWE-bench ベンチマークで MiniMax M2.5（80.2%）や Kimi K2.5（76.8%）が Claude Opus 4.6 に肉薄し、オープンソースモデルでクローズドモデルに匹敵する結果を出している。
