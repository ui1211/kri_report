# 2026-04-22 点群データ OSS 調査

## 要約
本レポートは、点群データの可視化、処理、画像やメタデータ連携に使える OSS を A4 1 枚相当に整理したものである。調査対象は Open3D、CloudCompare、Potree、PCL を中心に、用途別の補助候補として OpenDRIVE 系ツールも含めた。結論として、Python で解析や AI 連携まで見据えるなら Open3D、現場運用や大規模点群の手動確認なら CloudCompare、Web 配信やブラウザ閲覧を重視するなら Potree が有力である。単一ツールで全要件を満たすよりも、解析系と閲覧系を分けて選定する方が現実的である。

## 現状
点群 OSS は解析ライブラリ、デスクトップ可視化、Web ビューアの 3 系統に大別され、それぞれ Open3D、CloudCompare、Potree、PCL が代表的である。Open3D は Python/C++ 両対応で点群読み込み、幾何処理、描画、機械学習連携までを一通り備えるため、Python 主体の検証環境や AI 連携との親和性が高く、検証立ち上がりを早めることが可能である。一方、CloudCompare は測量やインフラ点検での利用実績が多く、大規模点群の比較、編集、確認作業に強いが、GPL ライセンスの扱いとプラグイン拡張の運用設計が必要となる。Potree は WebGL ベースでブラウザ閲覧に向くため、社内共有や遠隔確認を前提にした構成で共有性が優れるが、WebGL 環境や配信基盤の準備が前提となり解析用途の中核にはしにくい。PCL は C++ 中心の定番ライブラリだが、依存関係と学習コストが重く導入ハードルが高まる。

## 課題
点群処理では可視化だけでなく、メタデータ保持、画像連携、変換時の情報欠落対策まで考える必要があるため、単一ツール万能主義では要件網羅が難しくなる。大規模点群ではメモリ消費、GPU 活用、LOD 制御の有無が運用性を大きく左右し、Python 主体の検証環境では Open3D が有力だが C++ ビルドや GPU 環境差分が導入負荷になる。CloudCompare は実務向きだが GPL ライセンスの扱いとプラグイン拡張の運用設計が必要であり、Potree は共有性に優れる一方 WebGL 環境や配信基盤の準備が前提となり解析用途の中核にはしにくい。つまり、解析系と閲覧系を分離せず単一ツールに依存すると、ライセンス、性能、運用負荷のトレードオフを管理しにくくなる。

## 解決
解析基盤は Open3D を第一候補とし、Python から点群入出力、前処理、可視化、ML 連携までを一元化することで、Python チームが試作から AI 連携までを進めやすく、検証立ち上がりを早められる構成とする。現場確認や手動比較用途には CloudCompare を併用し、大容量点群の確認や比較を既存運用に近い形で実施できるため、現場部門との連携も円滑となる。社内共有や閲覧用途には Potree を採用して Web 配信用のビューアとして役割を分離し、解析環境と配信環境を分けることで構成を単純化する。メタデータや画像のひも付けは単一 OSS に依存せず、元データ管理側で属性設計を維持する方が安全であり、結果として推奨構成は Open3D と CloudCompare を基本形とし、ブラウザ共有が必要な案件のみ Potree を追加する方式を採用する。

## 価値
Python チームは Open3D を軸に試作から AI 連携まで進めやすく、検証立ち上がりを早められるため、開発サイクルの短縮と機能実装の効率化が図れる。現場部門は CloudCompare により、大容量点群の確認や比較を既存運用に近い形で実施できるため、導入抵抗を下げつつ現場実装の定着を促進できる。閲覧共有を Potree に切り出すことで解析環境と配信環境を分離し、構成を単純化するため、複数環境管理の負荷を軽減できる。単一ツール万能主義を避けることでライセンス、性能、運用負荷のトレードオフを管理しやすく、結果として用途別に OSS を組み合わせる方針が、短期検証と将来拡張の両面で最も実務的である。
