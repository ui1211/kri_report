# 16GB VRAM 向け LLM モデル

## 要約
16GB VRAM 環境では「完全 GPU 収容モデル」と「CPU オフロード前提モデル」で体感性能が大きく分けるため、実運用では速度・品質・VRAM 収容性の三点最適化が重要。Glukhov 実測と Ollama Library を基準に、現状・課題・解決・価値の順で整理し、Plan/Act モード切替対応やコード特化モデルの採用を含めた 10 トピックの比較検討をまとめる。

## 現状
業界全体では 16GB VRAM で Q4 量子化で 8B〜20B 級の一部が「100% GPU 収容」で実用速度が出やすく、24B+ は CPU オフロードが増える傾向がある。配布面では `llama3.1` や `deepseek-r1` や `qwen2.5-coder` や `gpt-oss` などがローカル実行の主要選択肢として定着している。また Cline は Plan/Act モードでモデルを切替可能であり、Plan は長文脈・推論に、Act は速度・コード修正能力が実運用上の分岐点となっている。実運用事例としては、RTX 4080 16GB の環境で `gpt-oss:20b` が 139.93 tok/s、`qwen3.5:9b` が 90.89 tok/s、`ministral-3:14b` が 70.13 tok/s の実測が報告されている。

## 課題
C1: VRAM 制約とオフロード劣化。16GB を超えると CPU/GPU 分割が増え、応答速度が急低下しやすい。C2: Plan/Act で必要能力が異なる。Plan は長文脈・推論、Act はコード生成・修正と低遅延が求められ、単一モデル最適化が難しい。C3: コーディング精度の評価軸が分散している。McEval や EvalPlus や LiveCodeBench や Aider など複数軸を跨いだ判断が必要。C4: 日本語実務での品質差。英語系ベンチ強者が日本語実務で必ずしも最適とは限らない。C5: モデル更新が速く、陳腐化しやすい。数か月単位で新系列・新蒸留版が出るため、比較表の鮮度維持が難しい。

## 解決
C1 対応は OSS で Q4_K_M を基準に 100% GPU 収容モデルを優先し、研究で小型化・蒸留・量子化で推論効率を上げ、実運用で実測で高速な `gpt-oss:20b` や `qwen3.5:9b` を優先する。C2 対応は OSS で Cline の Plan/Act 切替を前提にモデルを分離運用し、研究で長文脈維持と推論品質を両立する運用設計を提案する。C3 対応は OSS でコード特化モデルを明示的に採用し、研究で複数ベンチの複合評価で過学習・偏りを緩和する。C4 対応は OSS で日本語比率の高いモデルを選択肢に組み込み、研究で日本語評価データ拡張とタスク別再学習を考慮する。C5 対応は OSS で Ollama タグ更新・モデル差し替えを前提に運用し、研究で継続ベンチと差分追跡の自動化を視野に入れる。

## 価値
コスト削減としてクラウド API 常時利用からローカル実行へ寄せることで従量課金を圧縮できる。性能最適化では 16GB で完全 GPU 収容できるモデルは、重オフロード構成より大幅に高速になる。生産性向上では Plan/Act 分離で「設計品質」と「実装速度」を同時に最適化しやすい。リスク低減では日本語タスクを特化モデルで補完することで、実務での誤解釈リスクを低減できる。
