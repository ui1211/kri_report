# report-format

## Goal

入力情報を素材として扱い、以下を同時に満たす Markdown レポートへ再生成する。

- 本文を読みながら、その場でリンク先を確認できる
- 注釈、脚注、参照 ID を使わない
- 見出し、順序、ラベルを固定し、出力ゆらぎを最小化する
- PDF へ変換しても外部リンクが残りやすい構造を使う

## Operating Rules

- 入力の見出し構造は流用しない
- 必ず本スキルのテンプレートをそのまま使う
- セクション名、順序、箇条書きラベルを変更しない
- トピックは内容ベースで分割し、1トピックごとに独立した小節を作る
- 各トピックで最低 1 つ、本文リンクを付与する
- 数値、仕様、比較主張には本文リンクを付ける
- 根拠がない場合は推測せず「不明」と書く

## Link Rules

### Body link form

本文では次の形式を使う。

`[説明テキスト](https://example.com/path)`

要件:

- 通常の Markdown リンクだけを使う
- Markdown 脚注構文 `[^id]` は使わない
- `[参照: ...]` のような内部参照リンクは使わない
- Web ソースがある場合は `https://` または `http://` を使う
- ローカルファイルへの `file:///` リンクは PDF ビューワで無効化されやすいので使わない

### Source list form

末尾の「6. 参考リンク一覧」では、本文で使った URL を重複なしで列挙する。

```md
## 6. 参考リンク一覧
- [記事タイトル](https://example.com/path)
- [別記事タイトル](https://example.com/other)
```

要件:

- 本文で使った Web リンクだけを列挙する
- 同一 URL は 1 回だけ書く
- 注釈 ID やメタ情報は付けない

## Mandatory Output Contract

出力は必ず次の骨格を維持する。見出し文言を変えない。

```md
# レポート

## 1. メタ情報
- 作成日時: YYYY-MM-DD
- 入力概要:
- トピック数:

## 2. 概要（全体）
- 

## 3. トピック一覧
### {トピック名}
- 概要: [説明テキスト](https://example.com/path)
- 特徴:
  - 
- 主要スペック or 数値:
- 制約 or 注意点:

## 4. 比較まとめ
- 違い:
- 使い分け:
- 強み / 弱み:

## 5. 更新履歴
- YYYY-MM-DD: 初版

## 6. 参考リンク一覧
- [記事タイトル](https://example.com/path)
```

## Section-Specific Constraints

### 1. メタ情報

- `作成日時` は ISO 日付または日時で書く
- `入力概要` は 1 行で簡潔に書く
- `トピック数` は整数で書く

### 2. 概要（全体）

- 箇条書きは 1 行以上
- 全体像、比較軸、重要な差分を要約する

### 3. トピック一覧

- 各トピックは `### {トピック名}` で開始する
- 各トピックに次の 4 ラベルを必ず入れる
  - `- 概要:`
  - `- 特徴:`
  - `- 主要スペック or 数値:`
  - `- 制約 or 注意点:`
- `- 概要:` の行には本文リンクを最低 1 つ入れる
- 数値、仕様、比較主張の行にも可能な限り本文リンクを入れる
- `- 特徴:` の直下に少なくとも 1 つの子箇条書きを置く

### 4. 比較まとめ

- 次の 3 ラベルを必ず入れる
  - `- 違い:`
  - `- 使い分け:`
  - `- 強み / 弱み:`

### 5. 更新履歴

- 1 行以上書く
- 初版なら `- YYYY-MM-DD: 初版` を使う

### 6. 参考リンク一覧

- 箇条書きのみで書く
- 本文で使った URL を重複なしで並べる
- 本文で一度も使っていない URL を追加しない

## Execution Procedure

1. 入力からトピック候補、数値、差分、参照先 URL を抽出する
2. トピック粒度をそろえ、`## 3. トピック一覧` に展開する
3. 各トピックの主張に通常の本文リンクを埋め込む
4. `## 6. 参考リンク一覧` に本文で使った URL を重複なしで列挙する
5. 下記チェックリストを満たすまで修正する
6. 必要なら `scripts/validate_report_format.py` で機械検証する

## Pre-Submission Checklist

- 見出しは `# レポート` + `## 1` から `## 6` までの固定構造になっている
- `## 3. トピック一覧` に 1 件以上の `###` トピックがある
- 各トピックの `- 概要:` に通常の本文リンクがある
- 各トピックの `- 特徴:` の下に子箇条書きがある
- `## 4. 比較まとめ` に 3 ラベルすべてがある
- 注釈、脚注、参照 ID が含まれていない
- `## 6. 参考リンク一覧` が本文リンクと整合している

## Validation

機械検証が必要な場合は次を実行する。

```powershell
python skills/report-format/scripts/validate_report_format.py path\to\report.md
```

このスクリプトは次を検査する。

- 固定見出しの有無と順序
- トピック構造
- 本文中の通常リンク
- 注釈、脚注、内部参照リンクの混入
- 参考リンク一覧の重複と未使用
- 比較まとめの必須ラベル

## Prohibitions

- 自由形式の出力
- 見出し名の変更
- セクション順序の変更
- トピック節の省略
- Markdown 脚注構文の使用
- `[参照: ...]` のような注釈記法の使用
- `file:///` リンクの使用
